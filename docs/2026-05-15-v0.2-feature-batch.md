# attn v0.2 Feature Batch

Date: 2026-05-15
Status: Planned

## Context

attn v0.1.3 ships a single-compositor (niri-only) focus tracker with a Chromium-family browser reader, a daily-break reminder, and a quickshell popup that surfaces Today / Week views. Several adjacent features are either half-built or missing:

- Firefox-derived browsers (Firefox proper, Zen, LibreWolf, Floorp) are invisible to the daemon today because the browser reader is hardcoded to Chromium's `urls` / `visits` schema and WebKit-epoch timestamps.
- Focus sourcing reads `niri msg -j event-stream` directly. River, Hyprland, and Sway users can't run attn at all even though they expose comparable IPC.
- `daily_budget_secs` parses out of config and lands in `CategoryStatus.budget_secs` in `attn status --json`, but the quickshell popup ignores the field — so the configured budgets do nothing visible.
- Break-overdue currently only surfaces in the popup chip. If quickshell isn't focused/open, the user has no signal. The daemon already speaks D-Bus (`zbus` is in Cargo.toml) but has no notification path.
- Uncategorized apps and domains land in the database with `watched: false` and are filtered out of the popup. The user can't discover what they're spending time on outside the configured categories without editing config by hand.

This batch lands all five together as a v0.2 release. Each phase is independently shippable so we can interleave with hotfixes; phases 1, 2, and 3 can be implemented in parallel by separate agents because their file scopes barely overlap. Phase 4 depends on phase 3's budget data path. Phase 5 stands on its own.

## Goal

After v0.2:
1. attn runs on niri, river, Hyprland, and Sway with a single config switch (`focus_source.kind = "<wm>"`).
2. Firefox / Zen / LibreWolf / Floorp history is read with the same snapshot-copy guarantees as Chromium.
3. The Today panel renders per-category budget progress (filled vs remaining), tinting categories that exceed their budget.
4. The daemon emits desktop notifications on break-overdue and on budget-exceeded events, rate-limited to avoid notification spam.
5. The popup grows an "Other" drawer that lists uncategorized apps and domains by time spent, with a one-click "Add to category…" action that writes the user's `config.toml` and reloads the daemon.

Done = all five acceptance lists pass, `cargo test` is green, and a single end-to-end manual session on niri exercises each feature without regression.

## Non-goals

- **No Wayland-only enforcement**: we keep the door open for X11-only compositors but don't implement an X11 focus source in this batch.
- **No mobile / KDE Plasma / GNOME support**: the focus-source trait must accept future adapters but plasma-shell / gnome-shell integration is deferred.
- **No new browser families beyond Firefox-derived**: Safari (WebKit), Opera (now Chromium), Vivaldi (Chromium) are out. Chromium reader already covers Vivaldi/Opera via `app_ids` config.
- **No notification preferences UI**: notifications enable/disable lives in `config.toml`; we don't add toggles to the popup settings sheet in this batch.
- **No retroactive uncategorized data backfill**: the "Other" drawer reads from existing `app_intervals` / `domain_intervals`, but we don't rewrite historical category assignments when the user adds a new category later. The drawer simply shows fewer items afterward.
- **No `config.toml` schema migration tooling**: `attn init --merge` (added in 0.1.3) is already the upgrade path; we don't write a separate migrator.

## Confirmed parameters

- **Compositor matrix**: niri (existing), river, Hyprland, Sway. River uses `riverctl spawn` + the layer-shell focus protocol via `wayland-protocols` — but its focus signal is exposed via `wlr-foreign-toplevel-management-v1`, the same protocol Sway uses. Hyprland uses `hyprctl -j activewindow` + its socket event stream. Default `focus_source.kind = "auto"` detects via env var (`NIRI_SOCKET`, `HYPRLAND_INSTANCE_SIGNATURE`, `SWAYSOCK`, `WAYLAND_DISPLAY` + protocol probe).
- **Firefox schema**: `moz_places` + `moz_historyvisits` tables in `~/.mozilla/firefox/*/places.sqlite`. Zen lives at `~/.zen/*/places.sqlite`. LibreWolf: `~/.librewolf/*/places.sqlite`. Floorp: `~/.floorp/*/places.sqlite`. All share Firefox's PRTime (microseconds since Unix epoch — no offset needed) for `visit_date` and `last_visit_date`.
- **Budget UI tint**: category bar segment dims to 35% opacity when category seconds < budget, full opacity at budget threshold, amber/red mix (`#d97b6c` blended with category color at 60/40) when over budget.
- **Notification rate limit**: break-overdue notification fires once per overdue session (cleared when the user takes a break ≥ `breaks.min_break_secs`). Budget-exceeded notification fires once per category per local day (key: `<date>:budget_exceeded:<category>` in daemon `meta` table).
- **"Other" drawer threshold**: uncategorized items with ≥ `display.domains_min_seconds` (default 30s) for domains and ≥ 60s for apps are shown. Sub-threshold items are aggregated into a single "(below threshold)" row.
- **Categorize-action atomicity**: daemon writes config via temp file + rename (`tempfile::NamedTempFile::persist`). User edits made between read and write are detected via file mtime check and abort the operation with a "config changed on disk, retry" socket error.

## Hard constraints

- **No breaking config changes**: existing `~/.config/attn/config.toml` files from 0.1.x must continue to load without `--merge` and without `init`. New fields must be `#[serde(default)]`.
- **`attn status --json` schema is additive only**: existing fields keep their names and types. New fields (e.g., `uncategorized_apps`) appear as optional fields the popup can ignore.
- **Single Cargo binary**: no split into multiple crates. Compositor adapters live behind a `FocusSource` trait inside `src/main.rs` (or split into `src/focus/` modules if main.rs gets unwieldy; main.rs is ~4500 lines today, splitting is acceptable).
- **No tokio**: keep the existing thread-based runtime. zbus 4.x in blocking mode is fine.
- **Test gate**: every phase's new code paths have at least one unit test in the existing `#[cfg(test)] mod tests` block, mirroring the patterns already used for `compute_active_session_seconds`, `apply_user_config`, etc.
- **Tag bump**: phases land as individual commits on `main`. Version bump to `0.2.0` happens after all five phases are merged and exercised.

## Tech picks

- **`notify-rust@4`** — wraps `org.freedesktop.Notifications` over zbus. Already-pulled `zbus` could be used directly but `notify-rust` saves ~50 lines of boilerplate and handles capabilities/timeout/urgency.
- **`wayland-client@0.31`** — already implied by the existing `ext_idle_notify_v1` integration. Reused for `wlr-foreign-toplevel-management-v1` (river/sway focus).
- **Built-in `serde_json::Value`** — for Hyprland's `hyprctl -j` JSON parsing, matching the existing niri pattern in `query_focused_window()`.
- **Existing `rusqlite`** — Firefox `places.sqlite` is SQLite; no new dep.
- **`tempfile@3`** — atomic config writes (`NamedTempFile::persist`). Already pulled in via `rusqlite`'s test deps, promote to a direct dep.

## Phase 1 — Cross-compositor focus source abstraction

**Goal:** Lift the niri-specific focus loop behind a `FocusSource` trait and add river/sway (via `wlr-foreign-toplevel-management-v1`) and Hyprland (via `hyprctl -j event-stream`) adapters. Default to auto-detect; `focus_source.kind` overrides.

### Files to create
- `src/focus/mod.rs` — defines `FocusSource` trait, `FocusedWindow` struct (moved from main.rs:1743), `FocusSink` channel, `auto_detect()` env-probe function, and `dispatch(kind, sink)` entry called from daemon startup.
- `src/focus/niri.rs` — contains `run_focus_loop()`, `query_focused_window()`, `spawn_niri_event_stream()`, `is_focus_event()`, `handle_focus_change()` lifted from main.rs:1629-1740.
- `src/focus/hyprland.rs` — connects to `$XDG_RUNTIME_DIR/hypr/$HYPRLAND_INSTANCE_SIGNATURE/.socket2.sock` for events, calls `hyprctl -j activewindow` for state.
- `src/focus/wlr_toplevel.rs` — uses `wayland-client` to bind `zwlr_foreign_toplevel_manager_v1`, tracks `activated` state per toplevel, emits `FocusedWindow` on change. Reused for river and sway.

### Files to modify
- `src/main.rs` — replace direct `run_focus_loop` thread spawn (line ~1038) with `focus::dispatch(config.focus_source.kind, sink)`. Move `FocusedWindow`, `OpenAppInterval`, `handle_focus_change` out into `focus/mod.rs` (or keep in main and re-export — implementer's call).
- `config/runtime.toml` — add `[focus_source]` block:
  ```toml
  [focus_source]
  kind = "auto"   # auto | niri | hyprland | river | sway
  ```
- `config/default.toml` — regenerated by `tools/sync-default-config.sh`.
- `Cargo.toml` — add `wayland-client = "0.31"` and `wayland-protocols-wlr = { version = "0.3", features = ["client"] }`.
- `src/main.rs` (Config) — add `FocusSourceConfig { kind: String }` and wire through `apply_user_config`.

### Key signatures

### Acceptance
- `cargo build --release` succeeds with new modules.
- On a niri session, `attn daemon` behaves identically to 0.1.3 (existing tests pass, manual focus changes still produce `app_intervals` rows).
- On a Hyprland session, `attn daemon` opens/closes intervals as windows are focused. `attn status --json` reports a non-zero `tracked_seconds` after 60s of usage.
- On a Sway session (or river), `attn daemon` does the same via wlr-foreign-toplevel.
- `focus_source.kind = "niri"` forces the niri adapter even under Hyprland (returns error if niri socket missing).
- `attn doctor` reports the active focus source as part of its output.
- A new unit test asserts `auto_detect()` returns the expected name when `HYPRLAND_INSTANCE_SIGNATURE` is set in the test env.

### Risks
- **Risk**: `wlr-foreign-toplevel-management-v1` doesn't expose `app_id` on river before version 0.3 of the protocol. **Mitigation**: query `pid` and read `/proc/<pid>/comm` as fallback, matching the existing niri PID resolution.
- **Risk**: Hyprland's socket2 emits events as `event>>data` lines, not JSON. **Mitigation**: parse line-prefix manually; only `activewindow` and `activewindowv2` events matter — call `hyprctl -j activewindow` to refresh full state on trigger (mirrors niri pattern).
- **Risk**: Splitting main.rs may break the `include_str!("../config/default.toml")` path or test-mod scoping. **Mitigation**: keep `tests` mod in main.rs and use `pub(crate)` exports; verify with `cargo test` before opening the PR.

Firefox SQL (target query):
```sql
SELECT
    moz_historyvisits.visit_date,
    0 AS visit_duration,           -- Firefox does not store duration; default 0, daemon clips via focus intervals
    moz_places.url
FROM moz_historyvisits
JOIN moz_places ON moz_places.id = moz_historyvisits.place_id
WHERE moz_historyvisits.visit_date >= ?1
  AND moz_historyvisits.visit_date <= ?2
ORDER BY moz_historyvisits.visit_date ASC
```
PRTime is microseconds since Unix epoch — convert with `Utc.timestamp_micros(prtime)`.

### Acceptance
- A Firefox profile at `~/.mozilla/firefox/<id>.default-release/places.sqlite` produces `domain_intervals` rows after `attn daemon` runs for 60s with Firefox focused on `https://example.com`.
- Zen at `~/.zen/<id>/places.sqlite` does the same with `kind = "firefox"`.
- LibreWolf and Floorp paths resolve via glob; no manual config edit needed.
- A unit test loads a fixture `places.sqlite` (committed to `tests/fixtures/firefox-places.sqlite`, ~50 KB) and asserts `BrowserReader::read_visits` returns the expected visits.
- The snapshot-copy continues to work when Firefox is running with the DB open (WAL + SHM sidecars are copied alongside).
- `attn doctor` lists each Firefox-family profile under "browser DBs" and reports `ok`.

### Risks
- **Risk**: Firefox doesn't record visit_duration. **Mitigation**: visits are clipped to focused app intervals downstream — duration of 0 means the existing `insert_attributed_domain_intervals()` clips each visit to the focus window containing its timestamp. This matches how Chromium short visits are handled when the recorded duration is smaller than the focus window.
- **Risk**: Zen's `app_id` varies by build (Flatpak vs native). **Mitigation**: include all observed variants in `app_ids` and document adding new variants in README.
- **Risk**: places.sqlite has heavier WAL than Chromium. **Mitigation**: snapshot-copy already grabs `-wal` and `-shm`; verify with a Firefox session that uses WAL mode (default since Firefox 90+).

## Phase 4 — Desktop notifications for break-overdue + budget-exceeded

**Goal:** Daemon emits `org.freedesktop.Notifications` D-Bus calls when a break becomes overdue and when a category crosses its budget, each rate-limited.

### Files to create
- `src/notifications.rs` — `NotificationSink` trait, `DesktopNotifier` impl (notify-rust), `NullNotifier` for tests, `NotificationKind` enum, dedup keys.

### Files to modify
- `src/main.rs` — daemon main loop instantiates a `DesktopNotifier`; the break-overdue check (line ~2406) calls `notifier.notify(NotificationKind::BreakOverdue { active_secs })` once per overdue session, gated by a `notified_break_overdue` flag in the daemon state. Budget check fires inside the per-category status builder (line ~2374) and is gated by a `meta` table row keyed `notified_budget:<date>:<category>`.
- `config/runtime.toml` — add:
  ```toml
  [notifications]
  enabled = true
  break_overdue = true
  budget_exceeded = true
  ```
- `config/default.toml` — regenerated.
- `Cargo.toml` — add `notify-rust = "4"`.

### Key signatures
```rust
// src/notifications.rs
pub enum NotificationKind {
    BreakOverdue { active_secs: i64, overdue_secs: i64 },
    BudgetExceeded { category: String, seconds: i64, budget_secs: i64 },
}

pub trait NotificationSink: Send + Sync {
    fn notify(&self, kind: NotificationKind) -> Result<()>;
}

pub struct DesktopNotifier { app_name: &'static str }
pub struct NullNotifier;

impl DesktopNotifier {
    pub fn new() -> Self;
}

impl NotificationSink for DesktopNotifier { /* ... */ }
impl NotificationSink for NullNotifier { /* always Ok */ }
```

Dedup state lives in `meta`:
- `notified_break_overdue` = `"<session_started_at_rfc3339>"` — cleared on resume from break (PauseReason::Manual or Idle with duration ≥ min_break_secs).
- `notified_budget:<date>:<category>` = `"1"` — never cleared within the day; rotates with date.

### Acceptance
- Configure `[breaks] interval_secs = 60` and let the daemon run with no break. After 60s, a desktop notification appears titled "attn — take a break" with body `"focused for 1m, over by Xs"`.
- Taking a break (≥ `min_break_secs`) and then accumulating another overdue session fires the notification again.
- Setting `[budgets.scroll] daily_budget_secs = 30` and accruing 31s of scroll usage fires a single "attn — scroll budget reached (31s / 30s)" notification. Subsequent seconds the same day do not refire.
- `[notifications] enabled = false` suppresses all notifications.
- A unit test wires a `NullNotifier`-replacement (test sink that records calls) into the budget builder and asserts a single call per category per day across 100 iterations of the status builder.

### Risks
- **Risk**: notify-rust depends on a session bus; headless test environments may panic. **Mitigation**: `DesktopNotifier::notify` returns `Err` instead of panicking when the bus is unavailable; daemon logs the error and continues. CI uses `NullNotifier`.
- **Risk**: Notification spam if dedup state corrupts. **Mitigation**: the `meta` row insert is `INSERT OR REPLACE` so a corrupt key gets overwritten on next trip; manual reset via `sqlite3 ~/.local/share/attn/state.db "DELETE FROM meta WHERE key LIKE 'notified_%'"` documented in README troubleshooting.
- **Risk**: User runs attn on a system without a notification daemon (mako, dunst, etc.). **Mitigation**: `attn doctor` probes for `org.freedesktop.Notifications` on the session bus and reports if absent.

## Phase 5 — Uncategorized "Other" drawer + add-to-category action

**Goal:** Today panel grows an expandable "Other" section listing uncategorized apps and domains by time. Each row has an "Add to…" picker that writes the user's config.toml and reloads the daemon.

### Files to create
- *(none — extends existing modules)*

### Files to modify
- `src/main.rs` — extend `Status` with two new fields:
  ```rust
  uncategorized_apps: Vec<AppStatus>,
  uncategorized_domains: Vec<DomainStatus>,
  ```
  Populate at the same point Today's `apps` / `domains` are built; filter to `watched == false` and `seconds >= threshold` (apps ≥ 60s, domains ≥ 30s).
- `src/main.rs` — add new socket command `categorize` accepting `{ kind: "app" | "domain", id: String, category: String }`. Handler:
  1. Read current `~/.config/attn/config.toml` (capturing mtime).
  2. Parse into the `Config` struct.
  3. Insert `id` into `config.apps.watch[category]` (or `config.domains.watch[category]`), preserving order, dedup.
  4. Verify mtime hasn't changed (`fs::metadata`).
  5. Write to `<config>.tmp` and `rename` over the original.
  6. Trigger an in-process reload (same path as `attn reload`).
- `src/main.rs` — CLI: add `attn categorize --kind app --id firefox --category coding` so the action is scriptable.
- `quickshell/AttnPopup.qml` — Today view gets a new expandable section "Other" below Domains. Header is a clickable row that toggles `popup.otherExpanded`. Rows render via `AttnRow` reused; each has a `+` button that opens a small QML `Menu` of existing categories. Selecting one shells `attn categorize --kind <k> --id <id> --category <c>` and reloads after.
- `quickshell/AttnPopup.qml` — add `popup.uncategorizedApps()` / `popup.uncategorizedDomains()` getters and a small `categoryPicker` Menu component reused per row.

- Running `firefox` for 2 minutes with `firefox` NOT in any watch category causes the popup's "Other" drawer to list `firefox — 2m` once expanded.
- Clicking the `+` next to `firefox` and choosing "coding" writes the change to `~/.config/attn/config.toml` (verifiable with `grep firefox ~/.config/attn/config.toml`), reloads the daemon, and the row migrates from "Other" to the Apps list under coding within one poll cycle.
- `attn categorize --kind domain --id example.com --category reference` from the CLI does the same for a domain.
- Concurrent config edit (touch the file after the daemon reads but before it writes) returns a `config changed on disk` error to the popup; popup shows a toast and re-fetches.
- Below-threshold items (<60s apps, <30s domains) aggregate into a single `(<n> below threshold)` row that's read-only (no `+` button).
- A unit test exercises `handle_categorize` against a fixture config and asserts the resulting TOML preserves user comments and ordering for non-touched categories. (If preserving comments is impractical, the test asserts the value lands in the right list and existing entries are intact — comment loss is documented in the README.)

### Risks
- **Risk**: `toml::to_string_pretty` does not preserve comments. **Mitigation**: use `toml_edit` (already a transitive dep via `cargo`) for the round-trip; if added, document in tech picks. If `toml_edit` proves heavy, accept comment loss and document it in README upgrade notes — users editing comments need to be aware.
- **Risk**: User adds the app to a category that doesn't exist yet (typo). **Mitigation**: validate `category` against existing keys in `config.apps.watch` / `config.domains.watch`; reject unknown categories with a clear error. Allowing new categories is deferred to a future plan.
- **Risk**: Uncategorized list grows huge on a fresh system. **Mitigation**: cap to top 25 per panel; aggregate the rest under "below threshold". User can run `attn status --json | jq .uncategorized_apps` to see the full list.

## Open questions / decisions deferred to implementation

1. **Compositor auto-detect ordering**: when multiple env vars are set (rare but possible during Wayland session-switch), prefer `NIRI_SOCKET` > `HYPRLAND_INSTANCE_SIGNATURE` > `SWAYSOCK` > wlr-toplevel probe. Implementer can swap order if probing finds a more reliable signal.
2. **Firefox `visit_duration`**: we record 0 and rely on focus-interval clipping. If clipped intervals end up too short in practice (sub-second URLs), the implementer can heuristically extend each visit to the next visit's timestamp, mirroring `cap_visits_at_next_start`.
3. **Notification urgency**: notify-rust supports Low / Normal / Critical. Default to Normal for break-overdue, Low for budget-exceeded. Implementer can promote break-overdue to Critical if users miss it in testing.
4. **"Other" drawer placement**: below Domains by default. If popup height becomes uncomfortable, move behind a tab. Implementer's call after the first build.
5. **`toml_edit` adoption**: only if comment preservation passes review. Otherwise stick with `toml::to_string_pretty` and document the trade-off.
6. **River-specific quirks**: river may need `riverctl list-inputs` or similar for app_id resolution if `wlr-foreign-toplevel-management-v1` reports empty app_id. Implementer should test on a real river session and pick the cheapest reliable signal.

## Rollback plan

Each phase lands as its own commit; rollback is per-phase via `git revert <sha>`. Specifically:

1. **Phase 1**: revert restores the inline niri focus loop. Users on non-niri compositors regress to no-data, matching 0.1.3 behavior.
2. **Phase 2**: revert removes the Firefox readers and the browsers/ module; Chromium reader continues to work because the chromium-specific code is re-inlined.
3. **Phase 3**: revert restores the popup to ignoring `budget_secs`; daemon still emits the field, no schema regression.
4. **Phase 4**: revert removes notify-rust and the notifications.rs module; daemon stops calling the sink. Users lose desktop notifications but the popup chip continues to surface break-overdue.
5. **Phase 5**: revert removes the categorize socket command and the Other drawer; config files written by Phase 5 remain valid (they just look like manual edits to a 0.1.x daemon).

Config sections added (`[focus_source]`, `[notifications]`, expanded `[[browsers]]` entries) are `#[serde(default)]` and harmless to leave behind on rollback.

## Critical files for implementation

- `src/main.rs` — daemon entry, status builder, socket handlers, config types. The 4500-line dispatch hub every phase touches.
- `src/focus/mod.rs` (NEW) — focus-source dispatch. Touched by Phase 1 only but every other phase reads its `FocusedWindow` type.
- `src/browsers/mod.rs` (NEW) — browser-reader dispatch. Phase 2.
- `quickshell/AttnPopup.qml` — popup UI. Phases 3 and 5 reshape it.
- `config/runtime.toml` + `tools/sync-default-config.sh` — every phase that adds config sections must run the sync script before committing; CI gates on it.
